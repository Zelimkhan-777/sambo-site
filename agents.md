# AGENTS.md

## 1. Назначение проекта

Проект представляет собой современный адаптивный многостраничный сайт Федерации самбо Чеченской Республики.

Сайт должен выполнять информационную и представительскую функции:

* рассказывать о федерации;
* публиковать новости и результаты соревнований;
* представлять руководство и тренерский состав;
* размещать документы;
* публиковать информацию о грантах и программах поддержки;
* предоставлять официальные контактные данные.

Это полноценный сайт спортивной организации, а не одностраничный рекламный лендинг.

Подробные требования к страницам и контенту находятся в:

```text
docs/project-requirements.md
docs/content-sources.md
docs/implementation-plan.md
```

Перед выполнением задачи изучи только те документы, которые относятся к текущему этапу работы.

## 5. Правила выполнения задач

Перед изменениями:

1. Изучи существующую структуру проекта.
2. Прочитай относящиеся к задаче требования.
3. Проверь уже реализованные компоненты.
4. Не создавай дублирующие решения.

Во время работы:

* выполняй только текущую поставленную задачу;
* не реализуй следующие этапы без прямого запроса;
* не переписывай работающий код без необходимости;
* не меняй стек;
* не добавляй зависимости, если задачу можно решить существующими средствами;
* не создавай чрезмерно сложные абстракции;
* не останавливайся после составления плана, если задача предполагает изменение файлов;
* сохраняй совместимость с уже реализованными страницами;
* учитывай мобильные устройства с начала разработки.

После изменений:

* проверь приложение;
* запусти доступный линтер;
* запусти production-сборку;
* исправь ошибки, вызванные внесёнными изменениями;
* кратко перечисли изменённые файлы;
* кратко опиши результат;
* укажи, что не удалось проверить.

Минимальные проверки:

```bash
npm run lint
npm run build
```

## 7. Цветовая система

Основная палитра сайта — монохромная.

Используй:

* глубокий чёрный;
* чистый белый;
* нейтральный графитовый;
* холодный светло-серый;
* почти чёрный фон для тёмной темы;
* красный и синий как редкие акценты.

Красный и синий отсылают к цветам экипировки самбо.

Не используй одновременно большое количество красных и синих элементов на одном экране.

Цветные акценты не должны конкурировать с фотографиями и основным контентом.

Не используй чистый `#000000` для всех крупных поверхностей. Создавай визуальную глубину с помощью нескольких нейтральных оттенков.

Рекомендуемые базовые значения:

```css
:root {
  --background: #ffffff;
  --surface: #f5f5f5;
  --surface-strong: #ebebeb;

  --foreground: #0a0a0a;
  --muted-foreground: #737373;

  --border: #e5e5e5;
  --border-strong: #d4d4d4;

  --accent-red: #b91c1c;
  --accent-blue: #1d4ed8;
}

.dark {
  --background: #0a0a0a;
  --surface: #171717;
  --surface-strong: #222222;

  --foreground: #f5f5f5;
  --muted-foreground: #a3a3a3;

  --border: #2e2e2e;
  --border-strong: #404040;

  --accent-red: #dc2626;
  --accent-blue: #3b82f6;
}
```

Используй семантические CSS-переменные.

Не разбрасывай конкретные значения цветов хаотично по компонентам.

## 8. Светлая и тёмная темы

Реализуй полноценные светлую и тёмную темы.

Требования:

* тема сохраняется в `localStorage`;
* при первом запуске учитывается системная тема;
* переключение происходит плавно;
* обе темы имеют достаточную контрастность;
* тёмная тема не является простой инверсией светлой;
* переключение темы не вызывает заметного мерцания;
* каждый компонент корректно выглядит в обеих темах.

React Context разрешено использовать для управления темой.

Не используй глобальный state manager только ради переключения темы.

## 10. Работа с изображениями

Используй:

* локальные изображения;
* разрешённые материалы официальных источников;
* нейтральные placeholders.

Не подключай изображения напрямую из Instagram как постоянный production-источник.

Не используй случайные фотографии из поисковой выдачи без проверки.

Не утверждай, что на фотографии находится конкретный человек, если это не подтверждено официальной подписью источника.

Для каждой фотографии желательно хранить:

* локальный путь;
* `alt`;
* источник;
* событие;
* дату;
* предполагаемое назначение;
* статус разрешения;
* статус проверки.

Рекомендуемая структура данных:

```js
{
  id: "competition-01",
  src: null,
  alt: "Спортсмены во время соревнований по самбо",
  sourceUrl: "",
  sourceName: "",
  category: "competition",
  usageStatus: "requires-approval",
  isVerified: false
}
```

Возможные категории:

```text
hero
about
news
competition
training
coaches
unused
```

Если подходящего изображения нет:

* используй аккуратный placeholder;
* не растягивай слабое изображение;
* не используй сильно размытые фотографии в крупных секциях;
* не останавливай разработку страницы.

Для hero подходят преимущественно горизонтальные изображения с достаточным свободным пространством для текста.

Для карточек разрешается использовать фиксированное кадрирование.

Обязательно:

* задавай стабильное соотношение сторон;
* используй `object-cover`;
* добавляй `loading="lazy"` вне первого экрана;
* добавляй корректный `alt`;
* предотвращай скачки layout;
* предусматривай fallback при ошибке загрузки.

Не изменяй оригинальные файлы в `assets/images/raw`.

Оптимизированные версии сохраняй отдельно.

* достижения;
* даты;
* соревнования;

* пустое состояние;
* нейтральный placeholder;

## 13. React и состояние

Используй локальный state для:

* мобильного меню;
* поиска;
* фильтров;
* сортировки;
* пагинации;
* раскрывающихся элементов;
* модальных состояний.

Используй React Context только при наличии реальной глобальной необходимости, например для темы.

Не дублируй derived state.

Не сохраняй в state значения, которые можно вычислить из props или другого state.

Используй `useMemo` только тогда, когда:

* вычисление действительно заметное;
* результат нужен как стабильная зависимость;
* есть понятная причина оптимизации.

Не применяй `useMemo` и `useCallback` автоматически ко всем функциям и значениям.

Не добавляй Redux или Zustand без прямого запроса.

Не форматируй весь проект ради одного небольшого изменения.

Не удаляй пользовательский код без объяснимой причины.

Не меняй публичный API существующих компонентов, если этого не требует задача.

Предпочитай небольшие логические изменения.

После завершения предложи подходящее название коммита в формате:

```text
feat: add responsive application header
fix: prevent mobile menu scroll
refactor: extract news card component
style: refine dark theme surfaces
```

2. этот файл;
